# คู่มือการใช้งานระบบ ai_web_scrpping อย่างละเอียด (Comprehensive Manual)

---

## บทที่ 1 ภาพรวมโปรเจกต์

โปรเจกต์ **ai_web_scrpping** เป็นระบบ Data Pipeline อัตโนมัติสำหรับการรวบรวมข้อมูลเชิงพื้นที่ (Location Intelligence) เพื่อวิเคราะห์ศักยภาพของพื้นที่ต่างๆ ในประเทศไทย โดยดึงข้อมูลจากหลายแหล่ง เช่น OpenStreetMap, Google Maps และข้อมูลภาครัฐ

เป้าหมายหลักคือการสร้างชุดข้อมูลมาตรฐานที่สะอาด (Canonical Outputs) เช่น ตำแหน่งสถานที่สำคัญ (Landmarks), สิ่งอำนวยความสะดวก (Amenities) และโครงข่ายเส้นทางจราจร (Road Networks) ที่พร้อมใช้งานสำหรับงาน BI, แดชบอร์ด หรือนำไปวิเคราะห์ศักยภาพของอสังหาริมทรัพย์และทำเล

---

## บทที่ 2 โครงสร้างโฟลเดอร์

```text
ai_web_scrpping/
├── data/ 
│   ├── raw/          # ข้อมูลดิบที่ Scraper ดึงมา (มักเป็น JSON) ไม่แนะนำให้ใช้ตรงๆ
│   └── processed/    # ข้อมูลที่ Clean แล้ว พร้อมใช้งาน (มักเป็น CSV)
├── scrapers/         # Script สำหรับดึงข้อมูลจากแหล่งต่างๆ (OSM, DOPA, Google Maps)
├── utils/            # Logic วิเคราะห์ข้อมูล, ทำความสะอาด (Clean & Merge) และ Quality Gate
├── docs/             # เอกสารคู่มือและการตั้งค่าต่างๆ
├── main.py           # Entry point หลัก (Full Pipeline)
└── run_*.py          # สคริปต์แยกสำหรับรันเฉพาะเรื่อง เช่น ถนน หรือ landmark
```

> [!TIP]
> **กฎเหล็ก**: อย่ารันไฟล์ใน `scrapers/` หรือ `utils/` โดยตรงจากข้างในโฟลเดอร์ ให้รันสคริปต์ที่มีคำว่า `run_` นำหน้าจากโฟลเดอร์ Root (`C:\ai_web_scrpping`) เสมอ เพื่อป้องกันปัญหา Path หาย

## บทที่ 3 สิ่งที่ต้องเตรียมก่อนใช้งาน

1. **Python Environment**: 
   - ระบบนี้รันด้วย Python 3.13 ขึ้นไป
   - ควรเรียกใช้งานผ่าน Virtual Environment เสมอ (`.\venv\Scripts\Activate.ps1`)
2. **การติดตั้ง Playwright**: 
   - จำเป็นสำหรับใช้บอทเปิดหน้าเว็บเบราว์เซอร์อัตโนมัติ (เช่น เวลาหาพิกัดจาก Google Maps)
   - คำสั่งติดตั้ง: `python -m playwright install chromium`
3. **การตั้งค่าไฟล์ `.env`**:
   - จำเป็นต้องมีไฟล์ `.env` อยู่ที่ Root Folder โดยระบุ API Key ดังนี้:
     ```env
     BOT_API_KEY=your_key_here
     ```
   - *หมายเหตุ: หากต้องการเปิดใช้ระบบเชื่อมกับ AWS S3 อย่าลืมแก้ไขชื่อ Bucket ใน `main.py` หรือตัวแปร S3 ที่เกี่ยวข้อง*

## บทที่ 4 วิธีรัน Pipeline หลัก (main.py)

ไฟล์ `main.py` คือศูนย์รวมคำสั่ง สามารถรันเพื่ออัปเดตข้อมูลทั้งหมดได้ในคราวเดียว:
```bash
python main.py
```

เมื่อกดรัน ระบบจะแสดงเมนู (Interactive Menu) ให้เลือกดังนี้:
- **[1] รันทุกอย่าง (Full Pipeline)**
- **[2] ดึงเฉพาะ Landmarks (OSM + Google Maps Sync)**
- **[3] ดึงเฉพาะ Restaurants**
- **[4] เลือกเองทีละรายการ (Custom)**

การทำงานของ **Full Pipeline [1]** ครอบคลุมขั้นตอนเหล่านี้แบบอัตโนมัติ:
1. ดึงข้อมูลสินเชื่อธนาคาร
2. ดึงสถานที่สำคัญ (Landmarks) และสิ่งอำนวยความสะดวก 
3. เติมเต็มข้อมูลด้วย Google Maps (Sync & Enrich)
4. ทำความสะอาดข้อมูล (Clean & Deduplicate)
5. ตรวจสอบความสมบูรณ์ผ่าน Quality Gate และบันทึกลงโฟลเดอร์ `processed/`

---

## บทที่ 5 คำสั่งย่อยที่ใช้งานบ่อย

หากต้องการอัปเดตข้อมูลแค่บางส่วนโดยไม่ต้องรัน Full Pipeline ให้ใช้ Modular Scripts ตามนี้:

- `python run_landmarks_only.py` : โหลดและอัปเดตเฉพาะ Landmarks สำคัญในเขตจังหวัดเป้าหมาย
- `python run_roads_only.py` : โหลดข้อมูลและวิเคราะห์โครงข่ายถนน (Road Network) จาก OSM
- `python run_restaurants_only.py` : ดึงและอัปเดตเฉพาะร้านอาหาร
- `python run_property_road_access.py` : คำนวณระยะการเชื่อมต่อของที่ดินและอสังหาฯ ออกสู่ถนนสายหลัก
- `python run_google_road_enrichment.py` : ดึงชื่อถนนภาษาไทยจาก Google มาเติมเต็ม OSM

## บทที่ 6 Data Flow และสถาปัตยกรรมระบบ

1. **Ingestion (การรวบรวม)**: Scraper ดึงข้อมูลดิบจาก API (OSM, ทศท., Bank of Thailand) แล้วแปลงและเซฟเก็บไว้ในรูปแบบ JSON ที่โฟลเดอร์ `data/raw/`
2. **Processing (การประมวลผลและทำความสะอาด)**: โค้ดในโฟลเดอร์ `utils/` จะนำ JSON มารวม, ลบรายการสถานที่ซ้ำซ้อน, ตรวจสอบรูปแบบ (Format) และเติมฟิลด์ให้ครบ
3. **Enrichment (การเสริมข้อมูล)**: ในส่วนของสถานที่และถนนที่มีแต่พิกัด ระบบจะใช้ Playwright ควบคุม Chromium ให้ยิงไปหา Google Maps เพื่อเอาชื่อจริง พิกัดจริงที่แก้ไขแล้ว และเรตติ้งกลับมา
4. **Export (การส่งออกและจัดเก็บ)**: สุดท้าย ข้อมูลจะถูกแปลงโครงสร้างเป็นไฟล์ CSV เซฟลงใน `data/processed/` ให้พร้อมนำไปใช้เข้า Database หรือเชื่อมต่อ Dashboard ได้เลย

## บทที่ 7 Output และ Data Contract

ข้อมูลที่เสถียรและพร้อมนำไปใช้งานจริง (Canonical Outputs) จะอยู่ในโฟลเดอร์ `data/processed/` ได้แก่:

### 1. `landmarks_clean.csv`
ไฟล์เก็บข้อมูลจุดสังเกตสำคัญ
- **คอลัมน์สำคัญ**: `province`, `name`, `category`, `layer` (ระดับความสำคัญ), `lat`, `lon`

### 2. `roads_final.csv`
ไฟล์เก็บข้อมูลโครงข่ายถนนที่ตัด Detail ไม่จำเป็นออกหมดแล้ว
- **คอลัมน์สำคัญ**: `road_name` (ชื่อจาก OSM), `road_display_name` (ชื่อจริงหลังผ่านการ Enrich), `highway_type`, `length_km`

### 3. `bank_loans_clean.csv`
ไฟล์รวบรวมอัตราดอกเบี้ยและสินเชื่อล่าสุด
- **คอลัมน์สำคัญ**: `bank_name`, `interest_rate`, `updated_at`

> [!WARNING]
> ไม่ควรหยิบไฟล์ใดๆ ใน `data/raw/` ไปใช้วิเคราะห์ เนื่องจากข้อมูลเหล่านั้นเป็นแค่ Cache ที่ยังไม่สกัดตัวซ้ำซ้อนออก และโครงสร้างอาจเปลี่ยนแปลงตลอดเวลาจากการอัปเดต

## บทที่ 8 Quality Gate

**Quality Gate** เป็นขั้นตอนตรวจสอบว่าการทำงานของไปป์ไลน์นั้นราบรื่นและข้อมูลมีความสมบูรณ์เพียงพอก่อนที่จะอนุมัติให้อัปโหลดผลลัพธ์ (ทำงานอยู่ใน `utils/pipeline_quality.py`)

**สิ่งที่ Quality Gate จะทำการตรวจสอบ:**
- ข้อมูลสถานที่ (Landmarks) ต้องมีครบถ้วนทุกจังหวัดตาม Config
- ไฟล์ Output สำคัญต้องถูกสร้างจริง และไม่ใช่ไฟล์ว่างเปล่า (Empty File)
- โครงสร้างและจำนวนข้อมูลไม่ผิดปกติแบบฉับพลัน
- หากตรวจสอบพบข้อผิดพลาด จะเกิดแจ้งเตือน (Error Alert) เพื่อเบรกระบบไม่ให้ส่งข้อมูลขยะไปใช้งานต่อใน Downstream

## บทที่ 9 Troubleshooting

ปัญหาที่พบบ่อยและวิธีแก้:

### 1. ยิง Google API ขัดข้อง (REQUEST_DENIED)
- **สาเหตุ**: เกิดจากไม่มี API Key หรือติด Limit
- **วิธีแก้**: ตรวจสอบการผูกบัตรใน GCP (Billing), ตรวจหา `GOOGLE_MAPS_API_KEY` ใน `.env`, หรือเช็ก Quota / IP Restrictions.

### 2. เบราว์เซอร์พัง (Playwright Error)
- **สาเหตุ**: เครื่องยังไม่มีตัว Chromium 
- **วิธีแก้**: รันคำสั่ง `python -m playwright install chromium` ในคอมมานด์ไลน์

### 3. หาไฟล์ใน `data/raw/...` ไม่เจอ 
- **สาเหตุ**: รันสคริปต์ผิดโฟลเดอร์ 
- **วิธีแก้**: รันรันสคริปต์จากตำแหน่ง Root (`C:\ai_web_scrpping`) เสมอ

### 4. ได้ข้อมูลน้อยผิดปกติ
- **สาเหตุ**: โดนแบนชั่วคราวจากเว็บต่างๆ
- **วิธีแก้**: รอประมาณ 5-10 นาที แล้วใช้ฟีเจอร์โหลดจาก Cache แทนการ Scrape ใหม่ทั้งหมด

## บทที่ 10 Best Practices

- **จัดการ Cache อย่างระมัดระวัง**: ไฟล์ในโฟลเดอร์ `raw/` อย่าง `geocode_cache.json` หรือ `google_road_name_cache.json` จะช่วยให้ไม่ต้องยิงเซิร์ฟเวอร์บ่อยๆ หากพบข้อมูลไหนที่ล้าหลัง ให้ลบไฟล์นี้ทิ้งเพื่อให้ระบบประมวลผลและดึงข้อมูลใหม่
- **ใช้ `road_display_name` เสมอ**: ในการวิเคราะห์ถนน ให้ใช้ฟิลด์นี้แทน `road_name` ธรรมดา เพื่อหลีกเลี่ยงชื่อถนนที่เป็น Tag ยึกยือจาก OSM เช่น "unnamed:12345"
- **ระวังการ Commit Data**: หมั่นตรวจ `.gitignore` ไม่ให้ทำการ Push ไฟล์ CSV หรือ JSON ขนาดใหญ่หลาย MB ขึ้น Git Repository เพราะจะทำให้ Repository บวมเกินไป

## บทที่ 11 ภาคผนวก

### การจัดเรียงความสำคัญของถนน (Highway Type Priority)

ตามโครงสร้างข้อมูลของ OpenStreetMap โปรเจกต์นี้จะทำการให้คะแนนความสำคัญของถนนเป็นตัวเลข (ตัวเลขน้อย = สำคัญมาก / ลำดับต้นในการ Route เส้นทางเชื่อมต่อ)

- **`0`**: `trunk` (ทางหลวงสายประธาน ระดับเชื่อมภูมิภาค)
- **`1`**: `primary` (ทางหลวงสายหลัก ระดับเชื่อมจังหวัด)
- **`2`**: `secondary` (ทางหลวงสายรอง ระดับอำเภอ)
- **`3`**: `tertiary` (ทางหลวงท้องถิ่น ระดับตำบล/เขต)
- **`99` (ไม่มี Priority หลัก)**: ถนนสายย่อยต่างๆ เช่น `residential`, `unclassified`, `service`

*หมายเหตุ: ถนนแบบ `motorway` จะเป็นสายหลักระดับสูงเช่นกัน แต่เนื่องจากทางเข้าออกมักจะปิดกั้น (Access Restricted) ระบบจึงแยกการวิเคราะห์การเข้าถึง `motorway` ออกจาก Local Routing เพื่อความสมจริง*
